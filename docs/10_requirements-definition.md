## 1. 目的・背景

- **目的**：やまがBASE㈱のB2B事業（経営研修を主役に、新規事業開発支援／地方創生事業支援）の「顔」となる単一LPを構築し、**「相談する」への送客**を最大化する。
- **背景**：方針転換により施設集客から「学び＋事業開発のパートナー」へ再定義（`decision-log.md` 2026-06-01）。将来コーポレートサイトへ拡張できる粒度で設計する。
- **設計原則**：「ワクワク（情緒）×稟議が通る根拠（理性）」の二重価値構造。

## 2. ゴール / KPI

| 区分 | 指標 | 備考 |

| 主要CV | 「相談する」フォーム送信／問い合わせ数 | 全CTAをここへ収束 |

> 数値目標値は運営とすり合わせ（`facts-and-figures.md`【要確認】）。

## 3. スコープ

- **In**：単一LP（トップ1ページ）、§6の全セクション、レスポンシブ、CTA導線、計測基盤、パフォーマンス計測、SEO/OGP、問い合わせ導線。
- **Out（初期）**：予約システム連携、多言語、CMS化、会員/認証、ブログ。将来拡張余地として `decision-log.md` に記録。
- **未決（要確認連動）**：Firebase利用可否、フォーム送信先、独自ドメイン、GA4計測ID（§12）。

## 4. 対象ユーザー・対応環境

- **中心読者**：熊本県内（および近隣）の経営者・経営企画・人材育成責任者。モバイル閲覧が多い前提（モバイルファースト必須）。
- **対応ブラウザ**（最新版および直近1メジャー）：
  - モバイル：iOS Safari、Android Chrome（**主対象**）
  - デスクトップ：Chrome、Edge、Safari、Firefox
- **対応OS**：iOS 16+／Android 10+／Windows 10+／macOS 12+
- **非対応**：IE。レガシーブラウザはグレースフルデグレード（コンテンツは読める）。

## 7. レスポンシブ要件（モバイルファースト・必須）

### 7.1 ブレークポイント（Tailwind準拠）
| 名称 | 幅 | 主対象 |
|------|----|-------|
| base | 〜639px | **モバイル（最優先設計）** |
| sm | 640px〜 | 大型スマホ |
| md | 768px〜 | タブレット |
| lg | 1024px〜 | ノートPC |
| xl | 1280px〜 | デスクトップ |

### 7.2 セクション別の可変挙動
- **ヘッダー**：md以上で横並びアンカー＋右端CTA。base/smでロゴ＋ハンバーガー（shadcn `Sheet`）。
- **ヒーロー**：H1は `clamp()` で可変（例 28→44px）。ビジュアルはモバイル縦持ち→md以上で横並び/背景化。
- **実績バッジ**：モバイルは2列折返し→md以上で横一列。
- **3つの入口／2枠／FEATURE**：base 1カラム → md 2 → lg 3（入口）/ 2（その他）。
- **学びの場2軸**：base縦積み → md横並び（対等2カラム）。
- **フロー図**：base縦ステップ → md横並び＋矢印。
- **FAQ**：全幅アコーディオン。
- **追従CTA**：base/smのみ表示（画面下部固定、`safe-area-inset` 考慮）。md以上は非表示。
- **画像**：`next/image` で `sizes` を適切指定し、デバイス幅に応じた配信。

### 7.3 タッチ・操作
- タップ領域 **最小48×48px**（CTA・ナビ・アコーディオン）。
- ホバー前提のUIを作らない（タッチで完結）。
- 横スクロール（overflow-x）を発生させない。

### 7.4 検証ビューポート（受け入れ対象）
**375 / 390 / 414 / 768 / 1024 / 1280 / 1440px**。実機：iPhone（Safari）・Android（Chrome）各1。
- 検証手段：Chrome DevTools デバイスエミュレーション＋実機１台以上。スクリーンショット差分は `mockup/` の手法（Chromeヘッドレス）を流用可。

## 8. パフォーマンス要件・計測（必須）

### 8.1 目標値（モバイル・中位端末/4G相当）
| 指標 | 目標 |
|------|------|
| LCP（Largest Contentful Paint） | **< 2.5s**（良好） |
| INP（Interaction to Next Paint） | **< 200ms** |
| CLS（Cumulative Layout Shift） | **< 0.1** |
| TTFB | < 0.8s |
| Lighthouse Performance（モバイル） | **≥ 90** |
| Lighthouse Accessibility / Best Practices / SEO | **各 ≥ 95** |
| 初期JS（First Load JS, gzip） | **≤ 130KB** を目安（超過時は要相談） |

### 8.2 達成のための実装要件
- 静的化優先（`next build` で静的プリレンダー。プレフライト済みで全ルート静的）。
- 画像：`next/image`（AVIF/WebP・`loading="lazy"`・適切な `sizes`/`width`/`height` でCLS防止）。ヒーロー画像は `priority`。
- フォント：`next/font`（Inter / Noto Sans JP、`display:swap`、サブセット）。既に配線済み。
- 重量級ライブラリ・アニメーションライブラリを安易に足さない。装飾は CSS/軽量SVG。
- レイアウトシフト対策：画像・埋め込み・バッジに寸法指定。Webフォントのフォールバック整合。
- サードパーティ（地図・YouTube等）は遅延読込／クリック後ロード（ファサード）。
- コード分割：下部セクションの重い要素は必要に応じ動的import。

### 8.3 計測基盤（「計測できること」を要件として担保）
1. **Web Vitals 実測**：`web-vitals` を導入し、LCP/INP/CLS/TTFB/FCP を収集。送信先は GA4（`web_vitals` カスタムイベント）。Vercelデプロイ時は **Speed Insights / Analytics** も併用可。
2. **GA4**：`<head>` に計測タグのフック（`<!-- analytics: GA4 -->`）を用意し、計測ID確定後に挿入（`{{要確認:GA4計測ID}}`）。
3. **イベント計測**：全主要CTA・ナビ・相談会申込・FAQ開閉・スクロール深度に `data-cta` / `data-event` 属性を付与し、GA4イベント化（§9）。
4. **ビルド時計測**：Lighthouse（モバイル）でスコアを取得できる手順を整備（ローカル/CI）。バンドルサイズの可視化（`next build` 出力 or analyzer）。
5. **回帰防止**：公開前チェックリスト（§13）にパフォーマンス測定を含める。計測結果は数値で残す。

> 「計測ID未確定でも、計測の"仕組み"は先に組み込む」方針。確定後はIDを差すだけで稼働する状態にする。

## 9. 計測・イベント設計（GA4）

| イベント | 発火条件 | 主パラメータ |
|---------|---------|------------|
| `cta_click` | 各CTAクリック | `location`（hero/research/seminar/final 等）, `label` |
| `consult_open` | 相談/問い合わせ導線到達・クリック | `source` |
| `seminar_apply` | 説明＆相談会の申込クリック | — |
| `faq_toggle` | FAQ開閉 | `question` |
| `scroll_depth` | 25/50/75/100% 到達 | `percent` |
| `web_vitals` | CWV計測 | `metric`, `value`, `rating` |

- 実装：`data-*` 属性 ＋ 単一の薄いトラッキングユーティリティ。計測ID未設定時はノーオペ（本番のみ送信）。

## 10. デザイン要件（トンマナ・UI）

- **カラー**（CSS変数で一元管理。`globals.css` を本LPトンマナへ再設定）：
  - ベース：オフホワイト `#faf8f4` ／ 文字：ネイビーチャコール `#1f2630`
  - アクセント：灯籠アンバー `#e0852f` ／ 補助：グリーン `#3f6b5a`
  - ※正式ブランドガイド入手後に上書き（`{{要確認:ブランドカラーコード/ロゴデータ}}`）。
- **タイポ**：見出しは太字（900）で力強く、英字/数字は Inter。本文16px以上・行間ゆったり。FEATURE番号など"大きな数字"は装飾的に使う（②参照）。
- **モチーフ**：六角形（bY）をアクセントに。実績は月桂冠/バッジ風（①参照、ただし上品に）。
- **コンポーネント方針**：shadcn-ui（導入済み21点）を基盤に、`Accordion`（FAQ）・`Card`・`Button`・`Sheet`（モバイルナビ）・`Badge`・`Tabs`（必要なら学びの場）等を活用。独自装飾は最小CSSで。
- **画像**：実施・施設・講師の写真を主役に（許諾前提＝`{{要確認:LP用許諾写真}}`）。確定まで比率固定のプレースホルダ。
- **トーン**：派手さより信頼。情緒は写真と余白で出す。

## 12. 非機能・その他要件

- **アクセシビリティ**：見出し階層（h1は1つ）、コントラスト比4.5:1以上、alt必須、キーボード操作・フォーカス可視、`prefers-reduced-motion` 配慮。
- **SEO/OGP**：適切な `<title>`/`meta description`、OGP/Twitterカード画像、構造化データ（Organization）、`sitemap.xml`/`robots.txt`、正規URL。
- **解析/同意**：必要に応じ Cookie 同意の方針（GA4導入時）。`{{要確認:プライバシーポリシー/同意要否}}`。
- **デプロイ**：Vercel 想定（`{{要確認:ホスティング先・独自ドメイン}}`）。
- **フォーム**：送信先未確定のため、初期は既存 `/inquiry` への外部リンク。将来 Firebase/フォームツール連携の余地（`{{要確認:フォーム送信先}}`）。
- **ブラウザ計測の本番限定**：開発/プレビュー環境では計測を送信しない。
